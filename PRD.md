# PRD: stablenet-pad

## 1. 배경 및 목적

### 배경
Wemade의 StableNet(원화 스테이블코인 전용 EVM L1) 위에서 DEX를 구축하는 내부 파일럿 PoC 프로젝트.
기획팀이 직접 스마트컨트랙트 배포와 운영을 테스트넷에서 시험적으로 경험하고자 함.

### 목적
스마트컨트랙트를 StableNet 테스트넷에 안정적으로 배포하고, 결과물(주소, ABI)을 팀이 재사용할 수 있게 체계적으로 관리하는 시스템을 구축한다.

### 핵심 결과물
웹 대시보드에서 컨트랙트 파일을 업로드하면 배포 파이프라인이 실행되고, 결과(주소, ABI, 트랜잭션 해시)를 대시보드에서 확인하고 GitHub에 자동으로 기록하는 풀스택 시스템.

---

## 2. 목표 (Goals)

### MVP 목표
- 토큰 컨트랙트 + 풀 컨트랙트를 Proxy+Implementation 구조로 StableNet 테스트넷에 배포할 수 있다
- 배포 결과물(주소, ABI)이 자동으로 기록되고 GitHub에 관리된다
- 2번 팀(프론트엔드)이 별도 커뮤니케이션 없이 결과물을 바로 가져다 쓸 수 있다
- PoC 발표에서 라이브 데모 성공 — 실제 토큰 + 풀 컨트랙트를 웹 대시보드에서 임의로 배포하는 것까지

### 확장 목표 (MVP 이후)
- 배포 가능한 컨트랙트 종류 확대
- 비개발자도 사용할 수 있는 인터페이스 제공
- 컨트랙트 코드를 직접 작성할 수 있는 코딩 툴 제공
- 배포된 컨트랙트 상태 모니터링
- 긴급 정지 / 파라미터 변경 등 운영 액션 UI
- 배포 이력 및 변경 로그 시각화
- 멀티시그 기반 배포 승인 프로세스
- 메인넷 배포 지원
- 다른 EVM 체인으로 확장
- Claude Code 멀티에이전트 오케스트레이션 적용 (단일 에이전트 → 역할별 서브에이전트 분리)
- 에이전트 간 하네스 엔지니어링 적용 (에이전트 간 입출력 표준화, 테스트 자동화)

## 3. 범위 (Scope)

### In Scope (MVP)
- Hardhat + Foundry 기반 배포 스크립트 작성 및 실행
- Proxy + Implementation 패턴으로 토큰 + 풀 컨트랙트 배포
- StableNet 테스트넷 배포 (Chain ID: 8283)
- 배포 결과물(주소, ABI) JSON 파일 자동 기록
- GitHub을 통한 결과물 공유 (stablenet-pad, public)
- 재배포 시 결과물 자동 업데이트
- 컨트랙트 파일 업로드 → 배포 파이프라인 실행 웹 대시보드 (Next.js)
- MetaMask 연결을 통한 배포 서명 및 가스비(WKRC) 처리
- 배포 결과(주소, ABI, 트랜잭션 해시) 대시보드 시각화
- 로컬 실행 전용 (Hardhat 프로세스 실행 환경 필수)

### Out of Scope (MVP)
- 컨트랙트 코드 직접 작성 (2번 팀 담당)
- 운영 모니터링 및 긴급 정지
- 비개발자용 코딩 UI
- 메인넷 배포
- 멀티시그 승인 프로세스
- Vercel / 클라우드 호스팅 (로컬 전용)
- ERC-3643 컨트랙트 배포 (MVP 이후로 연기)

## 5. 사용자 및 사용 시나리오

### 사용자 유형
| 유형 | 설명 |
|---|---|
| 배포 담당자 | 컨트랙트 파일을 업로드하고 배포 파이프라인을 실행하는 사람 |
| 프론트엔드 담당자 | 배포 결과물(주소, ABI)을 가져다 wagmi/viem에 연결하는 사람 (2번 팀) |
| 확인 담당자 | 배포가 제대로 됐는지, 의도대로 동작하는지 검증하는 사람 (기획팀 전체) |

### 시나리오 1: 정상 배포 (Happy Path)
1. 배포 담당자가 웹 대시보드에 접속한다
2. 컨트랙트 유형을 선택한다:
   - **ERC20**: .sol 파일을 업로드하고 name/symbol/initialSupply를 입력한다
   - **LiquidityPool**: 파일 업로드 없이 tokenA/tokenB/fee 파라미터만 입력한다 (서버 번들 템플릿 자동 사용)
3. MetaMask를 연결하고 배포 실행 버튼을 누른다
4. 파이프라인이 실행되고 진행 상태가 실시간으로 표시된다
5. 배포 완료 후 주소, ABI, 트랜잭션 해시가 대시보드에 표시된다
6. 결과물이 GitHub에 자동으로 push된다
7. 프론트엔드 담당자가 GitHub에서 결과물을 pull해서 사용한다

### 시나리오 2: 배포 실패 시
1. 배포 파이프라인 실행 중 오류 발생
2. 대시보드에 실패 원인이 표시된다 (예: 가스비 부족, 컨트랙트 오류)
3. 배포 담당자가 원인을 확인하고 수정한다
4. 재시도 버튼으로 다시 실행한다

### 시나리오 3: 재배포 시
1. 기존 컨트랙트에 버그 발견 또는 수정 필요
2. 수정된 컨트랙트 파일을 다시 업로드한다
3. 배포 실행 → 새 주소 생성
4. 기존 JSON 파일이 자동으로 덮어쓰여진다
5. GitHub에 변경사항이 자동 push된다
6. 프론트엔드 담당자는 별도 작업 없이 최신 주소를 사용할 수 있다

## 6. 성공 지표

### 기술적 지표
- StableNet 테스트넷에 토큰 + 풀 컨트랙트가 정상 배포된다
- 배포 결과물(주소, ABI)이 JSON 파일로 자동 기록되고 GitHub에 push된다
- 2번 팀이 결과물을 받아 wagmi/viem에 연결하는 데 별도 커뮤니케이션이 필요 없다
- 재배포 시 결과물이 자동으로 업데이트된다
- 비개발자(기획팀)가 웹 대시보드에서 배포 결과를 확인할 수 있다

### 비즈니스 지표
- PoC 발표에서 라이브 데모 성공
- 실제 토큰 컨트랙트 + 풀 컨트랙트를 웹 대시보드에서 임의로 배포하는 것까지 시연

---

## 7. 제약 사항 및 가정

### 제약 사항
- 테스트넷 전용 (메인넷 배포는 Out of Scope)
- 컨트랙트 코드 자체는 2번 팀에서 제공
- 가스비(WKRC)는 MetaMask 지갑에서 수동 서명
- 로컬 전용 실행 (Vercel 등 서버리스 환경 미지원)
- `.env.local` 파일은 절대 GitHub에 커밋하지 않음 (GITHUB_TOKEN, 프라이빗 키 포함)
- `.gitignore`에 `.env.local`, `.env*.local` 반드시 포함

### 가정
- StableNet 테스트넷 RPC가 안정적으로 운영된다
- 2번 팀이 배포 결과물 형식(JSON)에 동의한다
- PoC 발표 전까지 개발 완료 가능하다
